## Claude:
Nothing in the code is causing this. Looking at `bear-hud.lua` and `focus.lua` — the only window operations they do are `win:raise()`, `win:focus()`, `win:minimize()`, and `win:setFrame()`. None of those set a window level.

This is **100% the "Assign to All Desktops" Mission Control option** you mentioned toggling. Here's the mechanism:

When an app is pinned to "All Desktops," macOS places its windows in the `CanJoinAllSpaces` collection layer. That layer renders above windows from apps assigned to specific desktops. So when you click another window to bring it forward, macOS's z-order treats "All Desktops" windows as a separate tier that stays visually on top. It's not "always on top" in the NSFloatingWindowLevel sense — it's a space-layer ordering quirk.

**Fix:**

1. Right-click Bear's icon in the Dock
2. **Options → Assign To → None** (or "This Desktop" if you want it pinned to your current space)

That should immediately restore normal z-order. The "hide all to get underneath" behavior is the classic symptom of this setting being on.
