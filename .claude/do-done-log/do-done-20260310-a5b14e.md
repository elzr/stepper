## Claude:
Your concern is valid. Here's the problem:

**After reboot with your dock connected**, `init()` sees 5 screens immediately → starts the periodic autosave (every 60s). You open apps, they land in default positions, and within a minute the autosave overwrites your good layout file. By the time you hit restore, it's gone.

The `onScreenChange` auto-restore won't trigger either, because `lastScreenCount` is already set to 5 in `init()` — there's no transition *to* 5.

A few options to fix this:

1. **Startup grace period** — don't autosave for N minutes after Hammerspoon starts, giving you time to open apps and restore manually
2. **Separate manual save** — keep a "pinned" layout file that autosave never touches, and restore from that
3. **Backup rotation** — autosave keeps the last N saves, so you can always roll back

I'd lean toward **option 2** — it's the simplest and most reliable. The manual save (fn+ctrl+alt+delete or whatever your binding is) writes to a different file than autosave. Restore reads from the manual save. Autosave keeps doing its thing for the wake-from-sleep case without risk of clobbering your reboot layout.

What approach appeals to you?

## Claude:
Looking at `restoreFromJSON` in `layout.lua:397-440`, it matches saved entries to live windows in three tiers:

1. **Exact title match** — saved `win:title()` == live `win:title()`
2. **40-char prefix match** — first 40 chars of title match (handles titles that change slightly, like Chrome tabs with loading indicators)
3. **Index fallback** — nth unmatched window for the same app name

All matching is scoped by **app name** (`win:application():name()`), so a Terminal window will never match a Chrome entry.

There are no persistent window IDs — macOS assigns new IDs after every app launch, so after a reboot they're all different. Title matching is the best we can do.

For your use case (mostly Terminal and Chrome windows), this means:
- **Terminal**: titles are typically the shell command or directory, so they match well if you have the same sessions open
- **Chrome**: titles are the page title of the active tab, which is very stable

The main risk is multiple windows of the same app with identical titles — those fall through to index fallback, which matches by order (whatever `hs.window.allWindows()` returns). That's nondeterministic but usually close enough.
