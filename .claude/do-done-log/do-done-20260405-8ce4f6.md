## Claude:
Good catch — the 43" TV + built-in is also 2 screens, which means **screen count alone no longer differentiates configs**. Both "samsung-portrait + TV + built-in" (3 screens) and "TV + built-in" (2 screens) are known configs, and crucially, count=2 can no longer be treated as transitional during dock ramp-up.

This changes the design significantly. We need a position-signature approach instead:

| Config | Count | Position signature |
|--------|-------|--------------------|
| quad-32 | 5 | `bottom+center+left+right+top` |
| samsung-portrait | 3 | `bottom+center+left` |
| tv-only | 2 | `bottom+center` |
| native | 1 | `bottom` |

The signature is derived from `buildScreenMap()` — the set of unique screen IDs mapped to position names. This handles the ramp-up problem too: during dock connect 1→2→3→4→5, the position signatures at each step are different from the final quad-32 signature, so only the final state matches.

But wait — at count=2 during ramp-up, the position map might also be `bottom+center` (if the first LG to connect lands above built-in). That's the same signature as tv-only. So we still need the stability window, but it's now even more important.

Let me update the plan with this 4th config and the position-signature approach.
