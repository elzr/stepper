Now I see the full picture. Here's what happened:

**Timeline:**
- **17:52:29** — Last good save: w12work@right
- **17:53:09** — Screen lock/display sleep (`loginwindow` warning)
- **17:53:29–18:31** — Periodic saves ran every minute but found **0 windows** (screen locked). The 0-window guard correctly prevented overwriting the good save.
- **18:20:26** — Screen watcher: 5→5. No restore triggered because the screen count never changed.
- **No wake event** — `screensDidWake` never fired (screen lock ≠ system sleep)
- **~18:31** — User unlocked. Windows reappeared, but macOS had shuffled them.
- **18:32:29** — First autosave after unlock: w12work@**top** (wrong!). The damage was done.

**Root cause:** The wake detection (`screensDidWake`) doesn't fire for screen lock/display sleep. The screen count stayed at 5 the whole time, so the reconnection path (5→non-5→5) didn't trigger either. When windows reappeared after 27 minutes of 0-window saves, the periodic autosave just saved the macOS-shuffled positions without any drift check.

**The fix:** Detect the transition from "0 windows visible" → "windows visible" in `autoSave()` and treat it like a wake event — run drift detection and restore instead of blindly saving.
