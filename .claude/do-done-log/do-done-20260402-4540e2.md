## Design: Per-Screen Window Position Memory

### Two-tier storage

| Tier | Key | Storage | Survives reload | Use case |
|------|-----|---------|-----------------|----------|
| **Session** | `winID` | In-memory | No | All windows — Kitty, Bloom, Chrome tabs |
| **Persistent** | `app + "\n" + title` | `data/screen-memory.json` | Yes | Bear notes, Chrome calendars, any stable-title window |

### Lookup order on arrival at screen

1. `sessionMemory[winID][targetPos]` — exact, this session
2. `persistentMemory[app+title][targetPos]` — cross-session fallback
3. Proportional mapping + edge snapping — current behavior (first visit)

### Save triggers

- **On departure** (every `moveToScreen` call): save current frame-relative-to-screen to both tiers
- **On undo** (in `moveToDisplay`): save departure screen position *before* restoring undo frame, so both directions are remembered
- **From autosave** (every 60s): update both tiers for all visible windows — catches manual drag/resize between explicit moves. Skipped when position protection is active (so macOS scrambles don't poison screen memory).
- **Disk write**: debounced, only when persistent memory has changed since last write

### What it replaces

- **`naturalSize[winID]`** — subsumed. Per-screen memory remembers the large frame on center and the shrunk frame on left.
- **`naturalPosition[winID]`** — subsumed. Per-screen memory remembers the actual position on each screen.
- **`displayUndo`** — **kept.** Undo is "go back to previous screen." Now it saves departure position first, so the screen you just left remembers where you were.

### Stored frame format

Relative to screen (fractions), same as `frameRel` in layout.lua:
```json
{"x": 0.1, "y": 0.05, "w": 0.8, "h": 0.9, "ts": 1711900000}
```
Survives resolution changes. Timestamp for pruning (30-day expiry).

### L006 integration

After a layout restore (wake/reconnection/manual), seed `sessionMemory` from the restored positions. So if wake-restore puts a Bear note on center, the user can move it to right and come back to center later with the correct position.

### Edge cases

| Scenario | What happens |
|----------|-------------|
| **Bear rename** | winID session memory still works. Old persistent entry orphaned, pruned in 30d. New title accumulates fresh entries on departure. |
| **Chrome tab detach** | New winID, no session memory. Persistent lookup by app+title may match if title is unique. |
| **Two windows, same app+title** | Session memory (by winID) keeps them separate. Persistent: last-writer-wins — acceptable since it's a fallback. |
| **Quick bounce** (move to center, immediately move away) | Saves the proportional-mapped position. User said this counts (A5). If they later carefully position on center, that overwrites it on next departure.

### Q1: Naming

**`h`** = height. `frameRel = {x, y, w, h}` is the standard Hammerspoon frame structure — `x` offset, `y` offset, `w` width, `h` height. Same four fields as `win:frame()`.

**`Rel`** = relative. The frame expressed as fractions of the screen, not pixels. So `{x=0.1, y=0.05, w=0.8, h=0.9}` means "10% from left edge, 5% from top, 80% of screen width, 90% of screen height." This is already the convention in `layout.lua` line 406 — same name, same format.

**`screenPos`** = the position name string: `"left"`, `"right"`, `"center"`, `"top"`, or `"bottom"`. In the JSON it's the inner key:
```json
{"Bear\nDaily Log": {"center": {frameRel...}, "right": {frameRel...}}}
```

### Q2: L006 collaboration

The plan does integrate, but let me spell out the two directions:

**L006 → screenmemory (seeding after restore)**: When L006 restores windows after wake/reconnection, it calls `seedFromRestore(win, screenPos, frameRel)` for each placed window. This immediately populates screen memory, so if you move a window to another screen right after wake, it already has a position to come back to. Without this, screen memory would be empty after every HS reload until you manually moved windows around.

**screenmemory → L006 (autosave sync)**: Every 60s when `layout.save()` runs, it also calls `updateFromLayout(entries)`, keeping screen memory current with any manual repositioning the user did between explicit moves.

**The deeper benefit**: L006's restore currently only knows "where was this window at the last autosave?" Screen memory knows "where has this window been on each screen, accumulated over time." Right now the plan seeds screen memory FROM L006 restores, but in the future, screen memory could be a fallback data SOURCE for L006 — e.g., when a window wasn't captured in the last autosave (it was on a different screen) but screen memory knows where it goes on the target screen. That's a natural evolution but I'd keep it out of scope for the first implementation.
