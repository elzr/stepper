## Claude:
The inconsistency you're seeing is a well-known macOS pain point with multiple displays, and there are a few concrete things to investigate.

## Why macOS "forgets" window positions

macOS identifies displays by their EDID serial number to associate saved layouts with specific screen configurations. With 5 displays — especially if any are behind a hub/dock — the main culprits are:

1. **Non-unique display identifiers**: Monitors behind Thunderbolt/USB-C hubs sometimes don't expose unique serials to macOS, so it can't distinguish them reliably between sessions.
2. **Connection order sensitivity**: If macOS enumerates displays in a different order on wake/boot, it reassigns which "slot" each display occupies, scrambling saved positions.
3. **macOS version regressions**: Monterey and later have had repeated bugs with display arrangement persistence after sleep/wake. Some releases are noticeably worse.

## Diagnostics

Check whether macOS sees all 5 displays as unique:
```bash
system_profiler SPDisplaysDataType | grep -E "(Display|Serial Number)"
```
If any show no serial number or duplicate serials, that's your problem.

## Things to try

- **Always connect displays in the same order and via the same ports** — especially for hub-connected displays
- **Check System Settings → Displays** after a bad wake cycle to see if arrangement scrambled
- **Reset display arrangement**: System Settings → Displays → Arrange, then drag everything back. macOS re-saves the layout.
- If you're on Sequoia, check if there's a relevant update — display memory regressions often get patched

## The Hammerspoon angle

Since you already have Hammerspoon, `hs.layout` is built-in and purpose-built for this. It lets you define named layouts and apply them with a hotkey:

```lua
local layout = {
  {"Safari",  nil, screen1, hs.layout.left50,  nil, nil},
  {"Terminal", nil, screen2, hs.layout.maximized, nil, nil},
  -- etc.
}
hs.hotkey.bind({"cmd","ctrl"}, "R", function() hs.layout.apply(layout) end)
```

This would be far more reliable than macOS's native behavior and fits naturally into Stepper's architecture. If you want to go that route I can help design a save/restore system that snapshots current window positions and replays them — either per-app or for all windows across all 5 screens.

```
LG HDR 4K  ← no serial
LG HDR 4K  ← no serial  
LG HDR 4K  ← no serial
LG HDR 4K  ← no serial
```

The hubs aren't passing through the EDID serial numbers from the monitors. So macOS has been identifying them purely by **which Thunderbolt port each hub is plugged into** — not by the displays themselves. That was working until something disrupted the enumeration order.

## Why it broke suddenly

The most likely triggers:
- **macOS update** — Sequoia has had several display management regressions; a recent update may have changed how it handles displays with duplicate/missing serials
- **Cable/hub reseated** — even plugging the same hub back into the same port can sometimes cause macOS to re-enumerate in a different order
- **Hub firmware update** — some hubs auto-update
- **Sleep/wake with a display briefly offline** — can cause macOS to re-number the whole arrangement

## Why it was "so good" before

When enumeration order was stable, macOS could use port position as a proxy for identity. It worked by accident. The moment that order changed or macOS's enumeration logic shifted, the whole house of cards fell.

## What you can try

1. **Check which hub is in which Thunderbolt port** — make sure both hubs are in the same ports as before (look at the physical port positions on your Mac)
2. **Cold boot with all displays connected** — gives macOS the best chance to enumerate consistently
3. **BetterDisplay** — has tooling specifically for display identification issues with hubs, can sometimes assign stable virtual IDs

But honestly: with 4 identical no-serial LG monitors behind two hubs, you're in permanently shaky territory. The Hammerspoon `hs.layout` approach would give you a one-keystroke "restore my layout" that's completely immune to how macOS feels about display identity that day.
