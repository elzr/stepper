## The Situation

Do-done logs are auto-generated markdown conversation records, one per session, stored in each project's `.claude/do-done-log/`. The logger hook creates them automatically — you have no control over their creation.

**The inconsistency causing the noise:** Most of your projects (mas2, topsight, fleet) have `.claude/` in `.gitignore`, so logs are invisible to git. But **stepper** (and elzr) don't — so git keeps showing them as untracked/modified files, which surfaces in every `git status` and commit flow.

## Your Options

**Option A: Add `.claude/` to stepper's `.gitignore`**
- Matches what you already do in mas2, topsight, fleet
- Logs still exist locally, still show up in the do-done dashboard
- You never see them in git status again
- Simple, one line

**Option B: Add `.claude/do-done-log/` to the global gitignore** (`~/.config/git/ignore`)
- Fixes this for every project at once, including future ones
- No per-project `.gitignore` edits needed
- Still lets you track other `.claude/` files (like `settings.local.json` which you already globally ignore)

**Option C: Track them in git deliberately**
- They become part of the repo history — a permanent record of how the code was built
- Downside: noisy diffs, grows repo size, conversation content in version control

## Recommendation

**Option B** — add `.claude/do-done-log/` to your global gitignore. It's the principled fix: do-done logs are already aggregated centrally in `~/.claude/do-done/` with the dashboard, so there's no value in also tracking them per-project in git. One global rule means you never think about this again in any project.
