## The Bug: `restoreFromJSON` line 622-624

```lua
    -- Mark as verified in position protection
    local key = protectionKey(entry.app, entry.title)
    protectedEntries[key] = nil     -- ← removes protection for the SAVED TITLE
```

After the restore moves a window, it removes protection for that entry's title — reasoning "I just put it in the right place, no need to protect it."

But with **index-fallback matching** (Tier 3), the restore matched the saved entry `w13days` to the wrong Bear window. Both `w12thoughts` and `w13days` used index-fallback because neither had an exact title match at restore time. The restore grabbed arbitrary unmatched Bear windows and moved them to the saved positions. But:

- The Bear window it moved to the left **wasn't actually showing w13days** — it was some other Bear note
- The actual w13days window remained on the right (where macOS had scattered it)
- Protection for `"Bear\nw13days..."` was deleted because the restore thought it was handled

60 seconds later, the autosave runs: finds the real w13days still on the right, looks up protection — nil. Records `w13days@right`. The corruption is now permanent.

## Other misplaced windows

Comparing manual save vs autosave, w13days is the **only Bear window that changed screens**. w12thoughts and apr2026 are missing from the autosave (likely closed by then), but no other window was placed on the wrong screen.

## The Fix

Don't remove protection for index-fallback matches — only for exact-title and prefix matches where we're confident the right window was moved:

```lua
    -- Mark as verified in position protection (only if match is reliable)
    if p.matchTier ~= "index-fallback" then
      local key = protectionKey(entry.app, entry.title)
      protectedEntries[key] = nil
    end
```

This way, if index-fallback moved the wrong window, protection stays active and the next autosave will substitute the correct saved position.

## Claude:
Now I see the deeper issue. The title never changed — the problem is the **matching algorithm itself is order-dependent**.

Here's what happened. The saved JSON lists Bear entries in this order:

1. `w12thoughts 16-22mar2026` (right) ← processed FIRST
2. `w13thoughts ...` (right)
3. `w13work ...` (right)
4. `w12work ...` (right)
5. `_mem NOW` (center)
6. `_meet tecnoMAS 25mar` (center)
7. **`w13days 23-29mar2026`** (left) ← processed SEVENTH

The matching runs in a **single pass** (line 547). For each saved entry, it tries exact-title → prefix → index-fallback, and marks the matched window as consumed (`matched[win:id()] = true`).

Here's the chain of failure:

1. **w12thoughts** is processed first. That note was closed (no window has this title). Exact-title fails. Prefix fails. Index-fallback kicks in — grabs **the first unmatched Bear window**, which happens to be the **real w13days window** (it's first in z-order). Marks it consumed.

2. Later, **w13days entry** is processed. Exact-title search loops through Bear windows looking for "w13days 23-29mar2026" — but `matched[w13days_window_id]` is already `true` from step 1. The `not matched[win:id()]` check (line 554) **skips over it**. Exact-title fails. Falls through to index-fallback, grabs some other random Bear window.

**Index-fallback for a stale entry STOLE the window that would have been w13days's exact match.** Then the protection removal (line 624) sealed the deal.

So there are actually **two bugs**:

1. **Matching is single-pass and greedy**: an earlier entry's index-fallback can consume a window that a later entry would match exactly. The algorithm should prefer exact matches over fuzzy matches.

2. **Protection removal trusts index-fallback**: even after a shaky match, protection is removed.

The proper fix for bug #1: **multi-pass matching**. First pass: only exact-title. Second pass: only prefix. Third pass: index-fallback. This way exact-title always wins regardless of JSON order.

Want me to implement both fixes?
